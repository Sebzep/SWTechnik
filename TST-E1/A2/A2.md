Antwort:
Keine Sorge, das klingt auf den ersten Blick komplizierter, als es ist! Die Aufgabe will von dir keinen perfekt fertigen, hochkomplexen Onlineshop. Der Dozent möchte stattdessen sehen, dass du die Arbeitsweise von TDD (Test-Driven Development) verstanden hast und diszipliniert anwendest.Das Entscheidende ist hier: Der Weg ist das Ziel. Dein Git-Commit-Verlauf (die History) ist dein eigentliches Abgabe-Dokument.Hier ist Schritt für Schritt erklärt, was genau von dir erwartet wird und wie du das am besten anstellst:Was bedeutet "Red $\rightarrow$ Green $\rightarrow$ Refactor" im Git-Log?Du darfst den Code nicht einfach runterschreiben. Du musst für jedes kleine Teil-Feature (z. B. "Produkt hinzufügen") exakt diesen Dreischritt gehen und jeden Schritt einzeln commiten:RED (Commit 1): Du schreibst zuerst einen Test für eine Funktion, die es noch gar nicht gibt. Wenn du den Test ausführst, muss er fehlschlagen (daher "Red").Beispiel-Commit-Nachricht: feat: add test for adding product (RED)GREEN (Commit 2): Du schreibst gerade so viel Code im Warenkorb, dass der Test gerade eben so besteht (daher "Green"). Keine eleganten Lösungen, einfach nur, dass der Test grün wird.Beispiel-Commit-Nachricht: feat: implement add product functionality (GREEN)REFACTOR (Commit 3): Jetzt räumst du den Code auf. Variablen schöner benennen, Duplikate entfernen, Struktur verbessern. Die Tests müssen danach immer noch grün sein.Beispiel-Commit-Nachricht: refactor: clean up cart structure (REFACTOR)Ein konkreter Fahrplan für deinen WarenkorbDa du nicht alles perfekt fertigstellen musst, pick dir 3–4 ganz simple Features heraus. Zum Beispiel:Feature 1: Der leere WarenkorbRed: Test prüft, ob ein neuer Warenkorb 0 Produkte hat. (Schlägt fehl, weil Klasse/Methode fehlt).Green: Erstelle die Klasse ShoppingCart und lass die Methode getCount() einfach hartcodiert 0 zurückgeben. (Test ist grün!).Refactor: Eventuell Syntax aufräumen.Feature 2: Ein Produkt hinzufügenRed: Test fügt ein Produkt hinzu und erwartet, dass getCount() jetzt 1 ist. (Schlägt fehl, weil es noch 0 zurückgibt).Green: Implementiere eine Liste im Warenkorb, füge das Produkt hinzu und gib die tatsächliche Größe der Liste zurück. (Test ist grün!).Refactor: Code lesbarer machen.Feature 3: Gesamtwert berechnenRed: Test fügt zwei Produkte mit Preisen (z. B. 10 € und 5 €) hinzu und erwartet als Gesamtsumme 15 €. (Schlägt fehl).Green: Schleife schreiben, die die Preise aufsummiert. (Test ist grün!).Wie gehst du jetzt praktisch vor?Projekt aufsetzen: Erstelle ein leeres Projekt in deiner bevorzugten Programmiersprache und ein Git-Repository (git init).Such dir das erste Mini-Feature: (z. B. Warenkorb initialisieren).Schreib den Test $\rightarrow$ Commit (Red).Schreib den Code $\rightarrow$ Commit (Green).Verbessere den Code $\rightarrow$ Commit (Refactor).Wiederhole das für die nächsten 2–3 kleinen Funktionen.⚠️ Wichtigster Stolperstein: Vergiss nicht zu commiten! Wenn du erst den Code schreibst und danach den Test, fliegst du auf. Der Dozent schaut sich die Zeitstempel (Timestamps) deiner Commits an. Man muss sehen, dass der Test-Commit vor dem Code-Commit kam.Am Ende lädst du einfach das Git-Repository (oder das Zip-Archiv inklusive des versteckten .git-Ordners) hoch. Genau diesen "Zick-Zack-Kurs" aus Tests und Code wollen sie sehen!
